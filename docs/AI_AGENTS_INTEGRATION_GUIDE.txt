# AI Agents Integration Guide for Saleor TMA Backend

> Opencode-compatible guide for integrating AI agents with the GraphQL backend powering the Telegram Mini App.

---

## 🎯 Overview

This guide describes how AI Agents should integrate with the GraphQL backend powering the Telegram Mini App. Follow specs under `specs/` (contract, flows, autotests) and the agent guides.

**Key Integration Points:**
- Target minimal GraphQL surface driven by frontend flows
- All GraphQL inputs must be validated server-side
- Use per-request auth context derived from Telegram Init Data
- Resolve errors into descriptive GraphQL errors

---

## 🔧 Prerequisites

Before integrating, ensure you have:

1. **Access** to the GraphQL endpoint (e.g., `https://<host>/graphql`)
2. **Known** Telegram Init Data payload format and header name (`Telegram-Init-Data`)
3. **Node/Go environments** aligning with repo's worker setup
4. **Wrangler** available if deploying tests against Cloudflare Workers
5. **Read contract specs**: `specs/01-api-contract.md`, `specs/02-interaction-flow.md`, `specs/03-autotests.md`

---

## 📡 API Surface and Contracts

Integration should target this minimal GraphQL surface:

### Queries (for lists and basic reads)
- Fetch restaurants, categories, dishes
- Cart retrieval

### Mutations (as defined in specs)
- Place orders
- Update cart (add/remove/update items)
- Clear cart

### Validation Requirements
- **Server-side validation** for all GraphQL inputs (use Zod or similar)
- Map validated inputs to Saleor’s required fields
- **Resolve errors** into descriptive GraphQL errors without leaking internal stacks
- **Per-request auth context** from Telegram Init Data attached to GraphQL context

---

## 🔐 Authentication & Context

### Mandatory Header
- All GraphQL calls **must** include: `Telegram-Init-Data: { ... }`

### Middleware Responsibilities
1. Verify and parse Telegram Init Data
2. Produce validated context object (e.g., `ctx.user`, `ctx.session`)
3. Make context available to resolvers

### Resolver Best Practices
- Rely on context rather than pulling auth data from headers repeatedly
- Context should contain user/session information for downstream logic

## 🔄 Usage Patterns (Frontend Integration)

### Typical Request Structure
```json
{
  "query": "query GetProducts { products { id name price } }",
  "variables": {}
}
```

### Required Headers
- `Content-Type: application/json`
- `Telegram-Init-Data: { ... }` (on every request)

### Example (Curl)
```bash
curl -X POST https://<host>/graphql \
  -H "Content-Type: application/json" \
  -H "Telegram-Init-Data: { ... }" \
  -d '{"query":"query GetProducts { products { id name price } }"}'
```

## 🖥️ Frontend Integration Details

The frontend (Telegram Mini App) consumes the GraphQL API for:

### Core Features
- Restaurant browsing
- Cart management  
- Order placement

### Authentication
- **All GraphQL operations** require `X-Telegram-Init-Data` header
- Header value comes from `window.Telegram.WebApp.initData`

### User Flow Mapping

#### Restaurant Browsing
- `queries/restaurants` → List all restaurants
- `queries/restaurantCategories` → Categories for selected restaurant
- `queries/categoryDishes` → Dishes in selected category

#### Cart Management
- `queries/cart` → Get current cart
- `mutations/addToCart` → Add item to cart
- `mutations/updateCartItem` → Update item quantity
- `mutations/removeCartItem` → Remove item from cart
- `mutations/clearCart` → Clear entire cart

#### Order Placement
- `mutations/placeOrder` → Create order with delivery location

### State Management
- Frontend state mirrors backend GraphQL types
- Enables seamless integration and data consistency

### Checkout Requirements
- **Either** geolocation **OR** Google Maps link (exactly one required)
- Backend validates delivery location input

### Error Handling
- Follows standardized GraphQL error shape
- `{ errors: [{ message, code, field? }] }`
- No internal stack traces exposed to frontend

## 📋 Data Formats and Validation

### Input Validation
- Use strict input schemas for all mutations
  - `PlaceOrderInput`
  - `DeliveryLocationInput`
  - Cart operation inputs
- Map fields to Saleor equivalents
- **Server-side validation required**
- Return structured errors for invalid input
- **Never leak sensitive data** in error messages

### Testing Approach
- Follow `specs/03-autotests.md` for critical journeys
- Run tests locally: `pnpm test` (or project test command)
- Ensure type checks pass
- Validate GraphQL surfaces against `specs/01-api-contract.md`

---

## 🔒 Security Considerations

### Telegram Init Data
- **Verify on every request**
- Extract identity securely in per-request context
- Never trust client-side data without verification

### Logging & Secrets
- Avoid revealing secrets in logs
- Use structured logging with redaction
- Sanitize output before logging

### Dependencies
- Keep dependencies up to date
- Audit for known vulnerabilities regularly
- Use lockfiles for reproducible builds
